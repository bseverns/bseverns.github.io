    <h1>Evidence Before Polish</h1>
    <p>A project becomes publicly legible through proof objects before it becomes polished. A routing diagram, latency capture, audio comparison, redacted screenshot, or maintenance photo can do more honest work than a finished-looking page with unsupported claims.</p>

    <h2>What this changes on the site</h2>
    <p>Atlas pages use controlled status language and name the next proof object when evidence is still incomplete. That is not an apology. It is part of the method.</p>

    <h2>Projects that make this visible</h2>
    <ul class="atlas-todo">
      <li><a href="/atlas/n/moarknobs42/">MOARkNOBS-42</a> keeps latency, mode feedback, and mapping evidence in view before enclosure polish.</li>
      <li><a href="/atlas/n/frzone/">frZone</a> needs a short capture showing threshold state and MIDI or OSC output at the same time.</li>
      <li><a href="/atlas/n/horizon/">Horizon</a> becomes legible through a before/after audio render before broader DSP claims.</li>
      <li><a href="/atlas/n/liverig/">live-rig</a> needs a redacted routing diagram before any full-system confidence language.</li>
    </ul>

    <h2>Public reading rule</h2>
    <p>If the proof is partial, the page should say so directly. If the repo contains deeper implementation truth, the page should link there and narrow the public claim.</p>
